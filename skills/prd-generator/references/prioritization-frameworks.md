# 기능 우선순위 결정 프레임워크

## RICE 스코어링

가장 널리 사용되는 정량적 우선순위 결정 방법.

### 공식

```
RICE Score = (Reach × Impact × Confidence) / Effort
```

### 각 요소 정의

| 요소 | 설명 | 측정 방법 |
|------|------|----------|
| **Reach** | 영향받는 사용자 수 | 분기당 예상 사용자 수 |
| **Impact** | 개인당 영향도 | 0.25 (최소) ~ 3 (최대) |
| **Confidence** | 추정 확신도 | 100% (높음), 80% (중간), 50% (낮음) |
| **Effort** | 개발 노력 | 인-월 (person-months) |

### Impact 점수 기준

| 점수 | 레벨 | 설명 |
|------|------|------|
| 3 | Massive | 핵심 가치 제안 변경 |
| 2 | High | 주요 지표 크게 개선 |
| 1 | Medium | 눈에 띄는 개선 |
| 0.5 | Low | 약간의 개선 |
| 0.25 | Minimal | 거의 인지 불가 |

### 예시

| 기능 | Reach | Impact | Confidence | Effort | RICE Score |
|------|-------|--------|------------|--------|------------|
| 소셜 로그인 | 5000 | 2 | 80% | 2 | 4000 |
| 다크모드 | 3000 | 0.5 | 100% | 1 | 1500 |
| AI 추천 | 2000 | 3 | 50% | 4 | 750 |

## MoSCoW 분류

정성적으로 기능을 네 가지 카테고리로 분류.

### 카테고리 정의

| 카테고리 | 설명 | 비율 가이드 |
|----------|------|------------|
| **Must have** | 없으면 출시 불가 | ~60% |
| **Should have** | 중요하지만 없어도 동작 | ~20% |
| **Could have** | 있으면 좋음 | ~15% |
| **Won't have** | 이번에는 안 함 | ~5% |

### 분류 질문 체크리스트

**Must have 판단**:
- [ ] 이 기능 없이 서비스가 동작하는가?
- [ ] 법적/규제 요구사항인가?
- [ ] 핵심 사용자 여정에 필수인가?

**Should have 판단**:
- [ ] 사용자 만족도에 큰 영향을 주는가?
- [ ] 경쟁사 대비 기본 기대치인가?
- [ ] 대안이 있는가?

**Could have 판단**:
- [ ] "nice to have"인가?
- [ ] 특정 사용자만 원하는가?
- [ ] 추후 추가해도 무방한가?

### 예시

| 기능 | 분류 | 이유 |
|------|------|------|
| 회원가입/로그인 | Must | 서비스 이용 필수 |
| 비밀번호 찾기 | Must | 기본 UX 필수 |
| 소셜 로그인 | Should | 전환율 향상 |
| 프로필 커스터마이징 | Could | 부가 기능 |
| 게이미피케이션 | Won't | MVP 범위 외 |

## Kano 모델

사용자 만족도 기준 기능 분류.

### 카테고리 정의

| 카테고리 | 있을 때 | 없을 때 | 예시 |
|----------|---------|---------|------|
| **Basic (기본)** | 당연함 | 불만족 | 로그인, 검색 |
| **Performance (성능)** | 만족↑ | 만족↓ | 속도, 용량 |
| **Delighter (감동)** | 감동 | 무관심 | 깜짝 기능 |
| **Indifferent (무관심)** | 무관심 | 무관심 | 내부 최적화 |
| **Reverse (역효과)** | 불만 | 만족 | 과도한 알림 |

### 기능별 질문 쌍

각 기능에 대해 두 가지 질문:
1. "이 기능이 **있다면** 어떻게 느끼시겠습니까?"
2. "이 기능이 **없다면** 어떻게 느끼시겠습니까?"

응답 옵션:
- 마음에 든다 (Like)
- 당연하다 (Expect)
- 상관없다 (Neutral)
- 참을 수 있다 (Live with)
- 마음에 안 든다 (Dislike)

## Value vs Effort 매트릭스

2x2 매트릭스로 빠른 의사결정.

```
        High Value
            │
   Quick    │   Big Bets
   Wins     │   (전략적)
  ──────────┼──────────
   Fill-ins │   Money Pit
   (후순위)  │   (재검토)
            │
        Low Value
    Low Effort ───── High Effort
```

### 각 사분면 전략

| 사분면 | 전략 | 우선순위 |
|--------|------|----------|
| Quick Wins | 즉시 실행 | 1순위 |
| Big Bets | 충분한 검토 후 실행 | 2순위 |
| Fill-ins | 여유 있을 때 | 3순위 |
| Money Pit | 재검토 또는 제외 | 보류 |

## ICE 스코어링

RICE의 간소화 버전.

### 공식

```
ICE Score = Impact × Confidence × Ease
```

### 각 요소 (1-10 스케일)

| 요소 | 설명 | 1점 | 10점 |
|------|------|-----|------|
| **Impact** | 목표 달성 기여도 | 거의 없음 | 게임 체인저 |
| **Confidence** | 성공 확신도 | 추측 | 데이터 기반 |
| **Ease** | 구현 용이성 | 수개월 | 수일 |

### 예시

| 기능 | Impact | Confidence | Ease | ICE Score |
|------|--------|------------|------|-----------|
| 온보딩 개선 | 8 | 7 | 6 | 336 |
| 새 결제 수단 | 5 | 9 | 4 | 180 |
| UI 리디자인 | 6 | 5 | 3 | 90 |

## 우선순위 결정 프로세스

### Step 1: 기능 목록 작성
- 브레인스토밍으로 모든 기능 나열
- 중복 제거 및 그룹화

### Step 2: 프레임워크 선택
| 상황 | 추천 프레임워크 |
|------|----------------|
| 데이터 있음, 정량적 결정 필요 | RICE |
| 빠른 분류 필요 | MoSCoW |
| 사용자 관점 중심 | Kano |
| 빠른 시각화 | Value/Effort Matrix |
| 간단한 점수화 | ICE |

### Step 3: 스코어링/분류
- 팀 합의로 점수 부여
- 이해관계자 피드백 반영

### Step 4: 결과 검증
- 상위 항목 합리성 검토
- 의존성 확인
- 리소스 현실성 검토

### Step 5: 로드맵 반영
- 우선순위에 따라 마일스톤 배치
- 의존성 고려하여 순서 조정
